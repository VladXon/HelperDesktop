# Политика безопасности

## Контрольный список реализованных мер

- **Хэширование паролей**: scrypt с параметрами `N=16384`, `r=8`, `p=1`, `keylen=64`. Соль генерируется через `crypto.randomBytes(16)` и хранится вместе с хэшем. Проверка — константное время (`crypto.timingSafeEqual`).
- **JWT и refresh-токены**: access-токен (15 минут) и refresh-токен (30 дней). Refresh-токены ротируются при каждом использовании. При попытке повторного использования старого refresh-токена срабатывает механизм reuse-detection: все refresh-токены пользователя отзываются.
- **Блокировка аккаунта**: после 5 неудачных попыток входа за 15 минут аккаунт блокируется на 30 минут. Счётчик сбрасывается при успешном входе.
- **Rate limiting**: общий лимит 100 запросов в минуту на IP в production; auth-эндпоинны (`/api/auth/login`, `/api/auth/register`, `/api/auth/refresh`, `/api/auth/password`) — 5 запросов в минуту на IP.
- **Internal API для бота**: все запросы от бота к серверу требуют заголовок `X-Bot-Secret`, значение которого сверяется с `BOT_SHARED_SECRET`. Запросы без заголовка или с неверным значением отклоняются.
- **Хранение токенов на клиенте**: токены сохраняются через Electron `safeStorage` (шифрование на уровне ОС, на macOS — Keychain, на Windows — DPAPI, на Linux — libsecret).
- **HTTP-заголовки**: в production подключается `helmet` с CSP и HSTS. В development helmet отключён для удобства отладки.
- **Аудит-лог**: все значимые действия (вход, выход, смена пароля, изменение email, неудачные попытки входа) пишутся в таблицу `audit_log` с указанием user_id, ip, user-agent и timestamp. Срок хранения — 90 дней, очистка выполняется фоновым заданием раз в сутки.

## Сообщение об уязвимостях

Если вы обнаружили уязвимость, пишите на **security@example.com**. Пожалуйста, не создавайте публичный issue и не публикуйте детали до выхода патча. В письме укажите:

- описание уязвимости и шаги для воспроизведения;
- потенциальное влияние (что может получить атакующий);
- версии затронутых компонентов;
- контакт для обратной связи.

Мы стремимся ответить в течение 5 рабочих дней. После подтверждения уязвимости согласуем сроки раскрытия и выпустим исправление.

## Безопасные практики для оператора

- Регулярно ротируйте `JWT_SECRET` и `BOT_SHARED_SECRET`. Минимум — 32 случайных символа.
- Используйте HTTPS на reverse-proxy (nginx + Let's Encrypt). HTTP-соединения между nginx и сервером допустимы только в пределах локальной сети.
- Ограничьте доступ к SSH на сервере по ключам; отключите вход по паролю.
- Включите UFW и разрешите только 22/tcp, 80/tcp, 443/tcp.
- Следите за свободным местом на диске: рост логов и базы данных должен быть в пределах ожидаемого. Бэкапы БД выполняются ежедневно (см. `RUNBOOK.md`).
- Срок хранения refresh-токенов — 30 дней. При компрометации устройства пользователь должен выйти из аккаунта через клиент или командой `/logout` в боте.
