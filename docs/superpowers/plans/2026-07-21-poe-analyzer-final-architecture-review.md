# PoE Analyzer 2.0 — Final Architecture Review

> **Date:** 2026-07-21
> **Status:** Pre-implementation audit of v2 migration plan
> **Auditor:** Senior TypeScript Architect + Game Engine Architect
> **Reference:** `docs/poe/path-of-building-analysis.md`, v2 plan

---

## 1. Executive Summary

The v2 plan is a substantial improvement over v1. StatRegistry, ConditionExpr tree, immutable ModDB, and step-based pipeline are all moving in the right direction. However, the plan has **8 structural problems** that would cause a rewrite within 12 months if not addressed now.

| # | Problem | Severity | Fix |
|---|---------|----------|-----|
| 1 | StatRegistry as hand-written object → unmaintainable at PoE scale | **CRITICAL** | Generated registry from game data, not hand-authored |
| 2 | 3-layer modifier model cannot represent buff chains (mod→buff→mod) | **HIGH** | Add `EffectLayer` for cascading modifiers |
| 3 | ConditionExpr is boolean-only; cannot express "recently", durations, numeric thresholds natively | **HIGH** | Add `TemporalCondition`, `NumericExpression` |
| 4 | Immutable snapshots create O(n²) copies per build comparison | **MEDIUM** | Use persistent data structures (immutable.js-style structural sharing) |
| 5 | Linear pipeline masks circular dependencies in buff/aura/skill resolution | **HIGH** | Two-pass pipeline with dependency declaration |
| 6 | Nested CalculationTrace explodes exponentially for DPS calculations | **HIGH** | Flatten trace to references; computed trace is a DAG, not a tree |
| 7 | Client-only calculation prevents server-side batch analysis | **MEDIUM** | Calculation engine must be isomorphic (works in Node.js workers) |
| 8 | Deleting all code at once stops all development for months | **HIGH** | Strangler pattern: new core alongside old, feature-flagged transition |

## 2. Problem 1: StatRegistry — God Object in Disguise

### The problem

The v2 plan proposes a hand-written:

```typescript
const StatRegistry = {
  defense: {
    life: statKey('defense.life'),
    energyShield: statKey('defense.energyShield'),
    // ... ~20 entries
  },
  offense: {
    physicalDamage: statKey('offense.physicalDamage', { isDamageType: true }),
    // ... ~15 entries
  },
  attribute: {
    strength: statKey('attribute.strength'),
    // ... ~3 entries
  },
} as const;
```

**Reality check:** Path of Exile has approximately **2,200+ internal stat IDs** (from `skill_stat_descriptions.txt` in game data, which PoB maps via `SkillStatMap.lua` — an 80 KB file). These include:

- `spell_maximum_base_fire_damage` → maps to `FireMin`, `FireMax`
- `active_skill_area_of_effect_radius_+%_final` → maps to `AreaOfEffect`
- `base_physical_damage_reduction_rating` → maps to `Armour`
- `physical_damage_taken_+%` → maps to `PhysicalDamageTaken`
- `support_gem_more_damage_with_bleeding_+%_final` → maps to bleeding more multiplier

Hand-writing a 2,200-entry registry is not viable. The maintenance burden is enormous — every league adds or renames stats. Every typo is a runtime bug disguised as correct because the stat key doesn't match game data.

### Solution: Generated Registry + Stat Mapping Layer

```typescript
// ── Layer 0: Raw game stat IDs (auto-generated from GGG data) ──

// Generated by a build script from game data files.
// Never hand-edited. Committed to repo for reproducibility.
// File: core/registry/generated/game-stat-ids.ts

export const GameStatIds = {
  /** Maximum life */
  MAXIMUM_LIFE: 2,
  /** Base maximum fire damage */
  SPELL_MAX_BASE_FIRE_DAMAGE: 16,
  /** Area of effect radius final modifier */
  ACTIVE_SKILL_AOE_RADIUS_PCT_FINAL: 33,
  /** Physical damage reduction rating */
  BASE_PHYSICAL_DAMAGE_REDUCTION: 72,
  // ... ~2,200 entries, auto-generated from extracted game data
} as const;

// ── Layer 1: Domain stat keys (architect-defined, stable) ──

// These are OUR domain concepts. They map to game stat IDs.
// New stats can be added here, but existing ones are versioned.
// File: core/registry/stat-keys.ts

export interface StatKey {
  readonly id: string;              // stable domain ID: 'defense.life'
  readonly displayName: string;     // "Maximum Life"
  readonly aggregation: AggregationKind;
  readonly modifiers: GameStatMapping[];  // which game stats this consumes
}

export type AggregationKind =
  | { kind: 'sum' }                // BASE + INC summed, MORE multiplied
  | { kind: 'product' }            // multipliers only
  | { kind: 'maximum' }            // max of all sources
  | { kind: 'override' }           // last-write override
  | { kind: 'flag' };              // boolean

interface GameStatMapping {
  gameStatId: number;              // from GameStatIds
  transform: 'identity' | 'inverse' | 'percent' | 'perMs';  // how to interpret
  scope?: 'local' | 'global' | 'enemy';  // where it applies
}

// ── Layer 2: Generated domain registry (build artifact) ──

// File: core/registry/generated/stat-registry.ts
// Generated by: `pnpm poe:generate-registry`
// Source: game data (skill_stat_descriptions + mod databases)

export const STAT_REGISTRY: Record<string, StatKey> = {
  'defense.life': {
    id: 'defense.life',
    displayName: 'Maximum Life',
    aggregation: { kind: 'sum' },
    modifiers: [
      { gameStatId: GameStatIds.MAXIMUM_LIFE, transform: 'identity', scope: 'global' },
    ],
  },
  'offense.fireDamage': {
    id: 'offense.fireDamage',
    displayName: 'Fire Damage',
    aggregation: { kind: 'sum' },
    modifiers: [
      { gameStatId: GameStatIds.SPELL_MAX_BASE_FIRE_DAMAGE, transform: 'identity' },
      { gameStatId: GameStatIds.FIRE_DAMAGE_PCT_INCREASED, transform: 'percent' },
      // ... multiple game stats may map to one domain key
    ],
  },
  // ... all entries generated from game data + mapping file
};

// ── Layer 3: Type-safe accessor (TypeScript magic) ──

// File: core/registry/stat-accessor.ts
// Uses the generated registry to build a type-safe proxy:

export const S = new Proxy(STAT_REGISTRY, {
  get(target, path: string) {
    const key = target[path];
    if (!key) throw new Error(`Unknown stat key: ${path}`);
    return key;
  },
}) as StatRegistryAccessor;

// Usage:
// S['defense.life']          → StatKey (compile-time known)
// queryModDB(snapshot, S['defense.life'], cfg)  → never misspelled
```

**Why this is better:**

1. **Game data drives the registry.** GGG changes stat IDs → regenerate → type errors show every affected place.
2. **Domain keys are stable.** We control `'defense.life'` — the underlying game stat ID can change without breaking our domain logic.
3. **Bidirectional mapping.** `gameStatId → domainKey` (for parsing) and `domainKey → gameStatId` (for queries).
4. **Not a god object.** The registry is a `Record<string, StatKey>`, not a nested namespace. No manual coordination. Generated, validated, committed.
5. **Extensible by data.** New league mechanics = new `domainKey → gameStatId` mappings in the mapping file. Regenerate. Engine doesn't change.

## 3. Problem 2: Modifier Model Cannot Represent Buff Chains

### The problem

The v2 plan has three layers:

```
ModifierDefinition → ModifierInstance → AppliedModifier
```

This works for simple modifiers like "+99 to maximum life". But PoE has **cascading modifiers** where applying one modifier causes another modifier to become active, which may cause a third:

```
"Gain Onslaught for 4 seconds on Kill"
    → OnKill event
    → Apply Buff: Onslaught (duration: 4s)
    → Onslaught buff grants: "20% increased Attack Speed, 20% increased Cast Speed, 20% increased Movement Speed"
    → These three modifiers are now active on the character
```

```
"Determination Aura grants: +2051 Armour"
    → Aura is active
    → Grants flat Armour modifier to player
    → But also: "Determination has 40% increased Aura Effect"
    → Actual armour granted: 2051 × 1.4 = 2871
    → This new armour contributes to: armour formula, armour→ES conversions, etc.
```

The 3-layer model collapses here because:
- `Onslaught` is both a **condition** (player state) and a **source of modifiers** (the 3 speed buffs)
- `Determination aura` depends on **aura effect** modifiers which are themselves modifiers
- The order matters: resolve aura effect first, then determine grant values, then apply those

### Solution: Add `EffectLayer` — Modifiers That Produce Modifiers

```typescript
// A modifier can trigger an effect, which applies more modifiers
interface ModifierDefinition {
  stats: ModStatEntry[];
  conditions?: ConditionExpr;
  multipliers?: MultiplierRef[];
  effects?: ModifierEffect[];       // NEW: this mod triggers effects
  source: ModSourceKind;
  rawText: string;
}

type ModifierEffect =
  | GrantBuffEffect                  // "Gain Onslaught for 4 seconds on Kill"
  | GrantAuraEffect                  // "Determination grants +2051 Armour"
  | GrantSkillEffect                 // "Trigger Level 20 Firestorm when you Crit"
  | ConvertStatEffect                // "Evasion Rating is converted to Armour"
  | OverrideStatEffect               // "Maximum Life becomes 1" (CI)
  | ConditionalModifierEffect;       // "Nearby enemies have -20% Fire Resistance"

interface GrantBuffEffect {
  kind: 'grantBuff';
  buffId: string;                    // 'onslaught', 'fortify', etc.
  duration: DurationExpr;            // base duration
  condition?: ConditionExpr;         // trigger condition (onKill, onHit, etc.)
  cooldown?: number;                 // ms cooldown
}

interface GrantAuraEffect {
  kind: 'grantAura';
  auraId: string;                    // 'determination', 'grace', etc.
  modifiers: ModifierDefinition[];   // what the aura grants
  reservationEfficiency: number;     // % reservation efficiency
  affectedByAuraEffect: boolean;     // does aura effect scale this?
}

// The pipeline resolves effects in order:
//
// 1. Collect all modifiers from all sources → ModDB snapshot
// 2. Evaluate conditions → which mods are active?
// 3. Resolve effects in dependency order:
//    a. Aura effect modifiers (they scale auras)
//    b. Aura grants (now with correct scaling)
//    c. Buff grants (conditional on state)
//    d. Conversion effects (Iron Reflexes, CI)
// 4. The effect produces NEW modifiers → merged back into ModDB
// 5. Re-evaluate conditions (new mods may change state)
// 6. Repeat until fixpoint (typically 2-3 iterations)
```

**Why this is necessary:**

The v2 plan treats "modifier application" as a one-time event. PoB treats it as an iterative process where modifiers produce more modifiers. Without `EffectLayer`, we cannot represent:
- Auras and their interactions with aura effect
- Buffs triggered by conditions (Onslaught on Kill, Fortify on Hit)
- Conversion mechanics (Iron Reflexes, CI, MoM)
- Conditional modifiers on enemies ("nearby enemies have -X% resistance")
- Flask effects (which are themselves buffs that grant modifiers)

## 4. Problem 3: ConditionExpr Cannot Express Temporal/Numeric Mechanics

### The problem

The v2 ConditionExpr is a boolean tree:

```
AND (enemyBurning ∧ skillFire) ∨ (frenzyCharges ≥ 3)
```

But PoE has mechanics that this cannot express:

**Temporal conditions:**
- "**Recently**" — true if event happened within last 4 seconds
- "**Every 4 seconds**" — pulses on a timer
- "**While casting**" — true during skill animation
- "**For 4 seconds after**" — true for duration after trigger

**Numeric conditions:**
- "**Up to** 30% more damage based on distance"
- "**Per** 10 Strength"
- "Effect is **proportional** to mana spent"

**Stack conditions:**
- "Per 50 Rage" (not a boolean — a multiplier of `floor(rage / 50)`)
- "For each Poison on enemy" (count-dependent)

### Solution: Separate Gates (boolean) from Scales (numeric)

```typescript
// ── Gate: determines IF the modifier is active ──
type ConditionExpr =
  // Boolean logic (unchanged from v2)
  | AndConditions
  | OrConditions
  | NotCondition
  // State conditions
  | StateCondition      // { var: 'Moving' }
  | ActorCondition      // { actor: 'enemy', var: 'Burning' }
  | SkillTagCondition   // { tag: 'attack' }
  | SlotCondition       // { slot: 'Helmet' }
  // Temporal conditions (NEW)
  | RecentlyCondition   // { event: 'onKill', window: 4000 }
  | TimerCondition      // { interval: 4000, phase: 'active' | 'inactive' }
  | DuringActionCondition; // { action: 'casting' | 'channeling' | 'moving' }

// ── Scale: determines HOW MUCH the modifier contributes ──
type ScaleExpr =
  | ConstantScale        // { value: 1 } — no scaling
  | ChargeScale          // { chargeKind: 'frenzy', perCharge: 1 }
  | StatScale            // { statKey: StatKey, per: 10 } — "per 10 Strength"
  | StackScale           // { source: 'poison' | 'rage' | 'enduranceCharge', perStack: 1 }
  | DistanceScale        // { min: 0, max: 70, perUnit: 0.143, unit: 'metres' }
  | ManaSpentScale;      // { window: 4000, perMana: 1 } — proportional to mana spent recently

// ── Combined: a modifier is active AND contributes value × scale ──
interface ModifierEffectiveness {
  gate: ConditionExpr;           // is it active?
  scale: ScaleExpr;              // how strong?
}
```

**Usage example:**

```typescript
// "20% more Damage per Frenzy Charge"
{
  stats: [{ statKey: S['offense.damage'], type: 'MORE', value: 20 }],
  effectiveness: {
    gate: { type: 'state', var: 'hasFrenzyCharge' },
    scale: { type: 'charge', chargeKind: 'frenzy', perCharge: 1 },
  },
}
// Contribution = 20 × frenzyChargeCount × (other MORE multipliers)

// "30% increased Damage while Leeching"
{
  stats: [{ statKey: S['offense.damage'], type: 'INC', value: 30 }],
  effectiveness: {
    gate: { type: 'state', var: 'isLeeching' },
    scale: { type: 'constant', value: 1 },
  },
}
// Active only when leeching. No scaling.

// "5% increased Attack Speed per 10 Dexterity"
{
  stats: [{ statKey: S['offense.attackSpeed'], type: 'INC', value: 5 }],
  effectiveness: {
    gate: undefined,  // always active
    scale: { type: 'stat', statKey: S['attribute.dexterity'], per: 10 },
  },
}
// Contribution = 5 × floor(dexterity / 10)
```

**What this enables:**

- **Gate/scale separation** — a modifier has two orthogonal axes: "am I active?" and "how strong?".
- **Formulas stay in data.** The engine knows how to evaluate `ChargeScale`, `StatScale`, etc. The data defines which one to use.
- **AI explainability.** `"20% more per Frenzy Charge × 3 charges = 60% more"` — traceable.
- **New league mechanics.** GGG adds a new scaling type (e.g., "per Sentinel Empowered") → add `SentinelScale` leaf to `ScaleExpr` union. Engine evaluator gets one new case. Done.

## 5. Problem 4: Immutable Snapshots — Copy Cost Understated

### The problem

The v2 plan proposes:

```typescript
interface ModDBSnapshot {
  readonly mods: ReadonlyMap<StatRegistry, readonly ModifierInstance[]>;
  readonly parent: ModDBSnapshot | null;
}

function addModToSnapshot(snapshot, instance): ModDBSnapshot {
  return {
    ...snapshot,
    mods: new Map(snapshot.mods).set(key, [...existing, instance]),  // SHALLOW COPY
  };
}
```

**Reality check for PoE scale:**

- A typical build has **500-2,000** modifier instances (items: ~50-100, tree: ~80-120, skills: ~200+, buffs: ~100+, auras: ~50+, config: ~20, base: ~60)
- The `mods` map has ~150 keys (one per stat)
- Each build comparison (e.g., "should I equip this new ring?") creates a new snapshot
- The UI may compare 5-10 items per second during item browsing
- `new Map(existing)` + spread for 150 keys = 150 allocations per snapshot

**The real problem isn't micro-allocations. It's the N-squared comparison pattern:**

```
For each of 10 candidate rings:
  Create snapshot_ring_i = addMod(snapshot_base, ring_i)
  Run full perform(snapshot_ring_i)
  Compare output
```

With immutable snapshots, each `addMod` copies the entire `mods` map. 10 rings = 10 full copies = 10 × 150 key operations. This is fine for 10, but not for 10,000 automated comparisons (the "Find Upgrades" feature).

### Solution: Persistent Data Structures (Structural Sharing)

Instead of copying the entire map, share the unchanged parts:

```typescript
// ── Persistent HashMap (structural sharing) ──

class PersistentMap<K, V> {
  private readonly root: TrieNode<K, V>;
  private readonly _size: number;

  // Returns a NEW map with the updated key. Unchanged keys share structure.
  set(key: K, value: V): PersistentMap<K, V> {
    // Path copy: clone only the trie path from root to key
    // All sibling subtrees are shared by reference (immutable, safe)
    const newRoot = this.root.assoc(hash(key), 0, key, value);
    return new PersistentMap(newRoot, value === undefined ? this._size - 1 : this._size + 1);
  }

  get(key: K): V | undefined {
    return this.root.find(hash(key), 0, key);
  }
}

// Usage:
const baseDB = buildModDB(build);           // 150 keys, 500-2000 mods
const ring1DB = baseDB.addMod(ring1Mod);    // ~32 trie nodes copied (log₂(150) path)
const ring2DB = baseDB.addMod(ring2Mod);    // ~32 trie nodes copied
// ring1DB and ring2DB share 95%+ of their structure with baseDB

// For iteration: we only iterate the mods bucket for one stat key at a time.
// For comparison: structural diff is near-instant (root comparison).
```

**Why persistent structures:**

| Operation | Immutable Copy (v2 plan) | Persistent Map (fixed) |
|-----------|--------------------------|------------------------|
| Add 1 mod to 150-key DB | O(n) full copy: 150 key ops | O(log n) path copy: ~5-8 ops |
| Compare 2 DBs (equal?) | O(n) deep compare | O(1): root reference equality |
| Memory: 10 build variants | 10 × full copies | 1 × base + 10 × small delta |
| Serialize for sharing | Full 150-key map | Delta + base hash |

**Implementation note:** We should use a library like `immutable` (immutable.js) or implement a minimal `PersistentMap` (~100 lines of TypeScript). The persistent trie is a well-understood data structure (Clojure, Scala, Immutable.js all use it).

**The parent chain stays.** The persistent map replaces `new Map(snapshot.mods)` shallow copy. The parent chain (`snapshot.parent`) remains for hierarchical queries (skill modList → player modDB → cached modDB).

## 6. Problem 5: Linear Pipeline Masks Circular Dependencies

### The problem

The v2 pipeline is a flat ordered list:

```
Step 1: MergeKeystones
Step 2: InitMinions
Step 3: ProcessFlasks
Step 4: ProcessBuffs
Step 5: ProcessCurses
Step 6: ProcessAilments
Step 7: ProcessExposure
Step 8: Defence
Step 9: DefenceEstimations
Step 10: Triggers
Step 11: Offence
```

But the real PoB pipeline has circular dependencies that it resolves through **two-pass iteration**:

```
Aura effect mods  ←── affects ──  Aura grants  ←── produces ──  More mods
                                                            │
                                                            └── modifies ──  Aura effect
```

```
Curse effect mods  ←── affects ──  Curse values  ←── produces ──  Enemy mods
                                                                     │
                                                                     └── modifies ──  Resistance
                                                                                          │
Enemy resistance  ←── consumed by ──  Exposure  ←── produced by ──  Player ailment effects
```

PoB breaks these cycles with a **two-pass approach**:

1. **Pass 1:** Gather all modifiers. Merge auras/buffs/curses with initial values.
2. **Post-pass:** Re-evaluate with corrected values (aura effect now applied → aura grants updated → modifiers re-merged).
3. **Pass 2:** Defence + Offence on the final merged state.

A flat 11-step pipeline cannot represent this.

### Solution: Dependency-Declared Pipeline with Fixpoint Iteration

```typescript
// ── Each pipeline phase declares what it reads and writes ──

interface PipelinePhase {
  readonly id: string;
  readonly reads: StatKey[];          // what stats this phase queries
  readonly writes: StatKey[];         // what stats this phase modifies
  readonly effects: EffectKind[];     // what side effects it produces
  execute(state: PipelineState): PipelineState;
}

type EffectKind =
  | 'modifiesModDB'       // phase adds/removes mods
  | 'modifiesConditions'  // phase changes condition state
  | 'modifiesMultipliers'; // phase changes multiplier counts

// ── The scheduler resolves phases in dependency order ──

// Phase ordering is NOT linear. It's computed from the dependency graph.
function schedulePhases(phases: PipelinePhase[]): PipelinePhase[][] {
  // 1. Build dependency graph: phase A depends on phase B if
  //    A.reads ∩ B.writes ≠ ∅ (A reads stats that B writes)
  // 2. Topological sort → ordered passes
  // 3. Detect cycles → group into fixpoint iteration passes

  // PoB equivalent passes:
  // Pass 1: [BaseStats, Items, Tree, Skills, Config]
  // Pass 2: [AuraEffect, AuraGrants, BuffGrants, CurseEffect, CurseGrants]
  //         → iterate until fixpoint (aura effect changes aura grants, etc.)
  // Pass 3: [FlaskGrants, AilmentCalculation, Exposure]
  // Pass 4: [Defence, DefenceEstimations]
  // Pass 5: [Triggers, Offence]
  return computePhaseOrder(phases);
}

// ── Fixpoint iteration resolves circular dependencies ──

function executePass(
  phases: PipelinePhase[],
  initialState: PipelineState,
): PipelineState {
  let state = initialState;
  let changed = true;
  let iterations = 0;

  // Iterate until no phase produces new changes (or max iterations)
  while (changed && iterations < 10) {
    changed = false;
    for (const phase of phases) {
      const before = state.modDB.fingerprint();  // structural hash
      state = phase.execute(state);
      if (state.modDB.fingerprint() !== before) {
        changed = true;  // phase modified modDB → re-run pass
      }
    }
    iterations++;
  }

  if (iterations >= 10) {
    // Fixpoint not reached — this is a data error, not an engine bug
    throw new FixpointError(`Phase pass did not converge after 10 iterations`);
  }

  return state;
}
```

**Why this matters:**

1. **Correctness.** Aura effect + aura grants is a known circular dependency. Our engine must resolve it, not sweep it under a linear pipeline.
2. **Extensibility.** A new league mechanic that creates a new circular dependency (e.g., "Mana Regen scales Ward Recovery, Ward scales Mana Regen") is handled by the same fixpoint iteration — just add the phases.
3. **Performance.** Each pass iterates only until fixpoint. Typical builds converge in 2-3 iterations. Expensive phases (Defence, Offence) run in the final pass only.
4. **Debuggability.** `phase.fingerprint()` before/after each iteration → identify which phase is oscillating.

## 7. Problem 6: Trace Explosion — Nested Tree vs Flat DAG

### The problem

The v2 plan's `CalculationTrace` is a recursive structure:

```typescript
interface CalculationTrace {
  formula: string;
  parameters: Record<string, CalculatedValue>;  // RECURSIVE
  sources: TraceEntry[];
}

// For DPS, this nests:
// DPS.trace.parameters.totalHit.trace.parameters.critAvg.trace.parameters.physBase.trace...
// A 10-depth recursion with 5 sources per level = 5^10 = ~10 million trace entries.
```

This is **combinatorially explosive** for complex calculations. A full DPS trace could be megabytes in memory and impossible to serialize to PostgreSQL.

### Solution: Trace as a Flattened DAG with References

```typescript
// ── Each computed value gets a unique ID ──

interface TraceNode {
  id: string;                        // unique ID: "calc:dps:fireball:totalHit"
  formula: string;                   // "{lifeBase} × (1 + {lifeInc}/100)"
  value: number;                     // the final computed value
  inputs: string[];                  // IDs of child trace nodes (NOT nested objects)
  modifierContributions: ModifierContribution[];  // individual modifier breakdowns
}

interface ModifierContribution {
  modifierId: string;                // reference to ModifierInstance.id
  contributed: number;               // how much this mod contributed
  computation: string;               // "99 base × (1 + 0.3 inc) = 128.7"
}

// ── The trace is a flat collection, not a tree ──

interface CalculationTrace {
  root: string;                      // root node ID
  nodes: Map<string, TraceNode>;     // ALL trace nodes, flat
  modifiers: Map<string, ModifierInstance>;  // ALL referenced modifiers
}

// For AI: resolve the DAG by walking node IDs
function explainTrace(trace: CalculationTrace, nodeId: string, depth: number): string {
  const node = trace.nodes.get(nodeId);
  if (!node) return '';

  let explanation = `${'  '.repeat(depth)}${nodeId}: ${node.value} = ${node.formula}\n`;

  for (const mod of node.modifierContributions) {
    const modInst = trace.modifiers.get(mod.modifierId)!;
    explanation += `${'  '.repeat(depth + 1)}← ${modInst.source.label}: ${mod.computation}\n`;
  }

  // Follow references (not nested objects — these are string IDs)
  for (const inputId of node.inputs) {
    explanation += explainTrace(trace, inputId, depth + 1);
  }

  return explanation;
}
```

**Why flat DAG over nested tree:**

| Aspect | Nested Tree (v2) | Flat DAG (fixed) |
|--------|-----------------|------------------|
| Same value referenced twice | Two copies in tree | One node, two references |
| Memory for 500-mod build | Exponential with formula depth | O(mods × statKeys) — bounded |
| Serialize to PostgreSQL | Deep nesting = large JSON | Flat array of nodes + mod references |
| AI access pattern | Walk tree recursively | Walk DAG with memoization (already computed) |
| Cacheable | No — tree is specific to this calculation | Yes — share nodes across similar calculations |

## 8. Problem 7: Client-Only Calculation Blocks Server Features

### The problem

The v2 plan locates all calculation in the Electron client:

```
Client: ModDB snapshots, CalcPerform, trace
Server: PostgreSQL (game data only)
```

This blocks:
- **Build sharing:** I export my build → you import it → you should see the SAME DPS numbers without re-running the calc on your machine.
- **Server-side validation:** "Is this DPS plausible?" (anti-fraud for build sharing)
- **Batch analysis:** "Rank all 10,000 public builds by EHP" (leaderboard feature)
- **AI analysis at scale:** Server-side analysis of multiple builds for meta intelligence

### Solution: Isomorphic Calculation Engine

The calculation engine must be **a pure library** that works identically in:
- Electron main process (Node.js)
- Electron renderer (via preload, for responsive UI)
- Server Node.js workers (for batch processing)
- Server API routes (for build sharing validation)

```typescript
// ── The engine depends on NOTHING platform-specific ──

// packages/poe-engine/ — NEW shared package
// Dependencies: none (pure TypeScript)
// Exports: ModDB, CalcPerform, all calculators, StatRegistry, trace system

// Usage in Electron client:
import { buildModDB, calcPerform } from '@helper/poe-engine';
const result = calcPerform(buildModDB(build));

// Usage in Express server:
import { buildModDB, calcPerform } from '@helper/poe-engine';
app.post('/api/builds/:id/analyze', async (req, res) => {
  const build = await loadBuild(req.params.id);
  const result = calcPerform(buildModDB(build));
  await storeAnalysis(req.params.id, result);
  res.json(result);
});

// Usage in server worker:
import { calcPerform, buildModDB } from '@helper/poe-engine';
workerPool.addJob('batchAnalyze', async (builds) => {
  return builds.map(b => ({
    id: b.id,
    dps: calcPerform(buildModDB(b)).offence.totalDps,
    ehp: calcPerform(buildModDB(b)).defence.effectiveHP,
  }));
});
```

**What this requires:**

1. **No Electron dependencies.** `safeStorage` calls must be injected via dependency interface, not imported directly.
2. **No `fs` access.** Game data is loaded from PostgreSQL on server, from IPC on client. The engine receives `GameData` as a parameter, never fetches it.
3. **No `window` access.** All UI concerns are outside the engine.
4. **Serializable output.** `CalculationTrace` uses string IDs, not class instances. `ModDBSnapshot` is a plain object structure.

**Boundary reframed:**

| Layer | Location | Framework Deps |
|-------|----------|---------------|
| `@helper/poe-engine` | Shared package | **None** — pure TypeScript |
| Data loaders | Client Electron | PostgreSQL via IPC |
| Build importer | Client Electron | XML parsing, safeStorage (secrets) |
| UI | Client renderer | React, Electron |
| API routes | Server | Express |
| Batch workers | Server | Node.js worker_threads |
| AI analysis | Server or Client | AI provider interface (injected) |

### The problem

The v2 plan deletes 40+ files before any new code exists:

**Feature flag strategy:**

```typescript
// apps/client/src/main/poe/engine/analyzer.ts

export async function analyzeBuild(build: Build): Promise<AnalysisResult> {
  if (featureFlags.ENABLE_V2_ENGINE) {
    // V2: new engine path
    const snapshot = buildModDBSnapshot(build);
    const output = calcPerform(snapshot);
    return toAnalysisResult(output);
  } else {
    // V1: legacy path (existing code, zero changes)
    const xml = await resolveXml(build);
    const dto = parsePoBXml(xml);
    return computeAnalysis(dto);
  }
}
```

**Migration phases with feature flags:**

| Phase | Flag | What ships |
|-------|------|-----------|
| 0 | `ENABLE_V2_STAT_REGISTRY` | StatRegistry generated, unit tests pass. Old engine unchanged. |
| 1 | `ENABLE_V2_CONDITIONS` | ConditionExpr engine, unit tests. Old engine unchanged. |
| 2 | `ENABLE_V2_MODIFIER` | Modifier model + parser adapted. Old engine unchanged. |
| 3 | `ENABLE_V2_MODDB` | ModDBSnapshot from old build data. **Integration test: old build → new snapshot → query → same life value as old engine.** |
| 4 | `ENABLE_V2_DEFENCE` | New calc-defence pipeline. **Integration test: old engine defence numbers = new engine defence numbers.** |
| 5 | `ENABLE_V2_OFFENCE` | New calc-offence pipeline. Same validation. |
| 6 | `ENABLE_V2_ENGINE` | Full pipeline. `analyzeBuild` uses new engine by default. Old `core/` becomes dead code. |
| 7 | `CLEANUP_OLD_ENGINE` | Delete `core/stats`, `core/calculation`, `core/calculators`, `core/rules`. |
| 8 | `CLEANUP_COMPAT` | Delete `compat/` adapters. |

**Why this matters:**

1. **Zero downtime.** The existing PoE analyzer keeps working through every phase.
2. **Regression safety.** At every phase, integration tests verify: `oldEngine.build(b) === newEngine.build(b)`.
3. **Incremental ship.** Each phase can be merged to `master` independently when its flag tests pass.
4. **Rollback easy.** If a phase breaks something, disable the flag. Old engine still works.
5. **Contributor-friendly.** Other developers can keep working on data layer, UI, server while engine migration proceeds.

## 10. Problem 9: Data-Driven Architecture — Escape Hatches Needed

### The problem

The v2 plan promises P8: "Adding a new keystone or gem should not require changes to the calculation engine." This is **unrealistic** for genuinely novel mechanics.

When PoB added Tinctures (3.25 Settlers league), they required:
- New CalcPerform logic for tincture mana burn stacks
- New CalcOffence logic for tincture weapon effect
- New flask deduplication logic (tinctures replace flasks)
- New reservation logic (mana burn is a reservation)

This was **~150 lines of new engine code**. No amount of data-driven architecture can anticipate a mechanic that didn't exist before.

**Honest assessment:** We can achieve **~80% data-driven**. The remaining 20% requires engine changes for genuinely novel mechanics. The architecture must make these changes **easy, isolated, and safe**.

### Solution: Plugin Architecture for New Mechanics

```typescript
// ── Engine provides hooks for novel mechanics ──

interface LeagueMechanic {
  readonly id: string;
  readonly leagueVersion: string;

  // Hook: modify the calculation environment before pipeline runs
  modifyEnv?(env: CalcEnv): CalcEnv;

  // Hook: insert custom phase into pipeline
  customPhase?: PipelinePhase;

  // Hook: modify how a specific stat is calculated
  statOverrides?: Map<StatKey, StatCalculator>;

  // Hook: add new stat keys to the registry
  newStats?: StatKey[];

  // Hook: add new condition types
  newConditionEvaluators?: Map<string, ConditionEvaluator>;
}

type StatCalculator = (
  snapshot: ModDBSnapshot,
  cfg: SkillConfig,
  baseValue: number,
) => number;

// The engine registers league mechanics at startup:
const leagueMechanics: LeagueMechanic[] = [
  tinctureMechanic,       // 3.25 Settlers
  crucibleMechanic,       // 3.21 Crucible
  sanctumMechanic,        // 3.20 Sanctum
  // ... one plugin per league with novel mechanics
];

// Each league plugin is in its own file:
// engine/leagues/tincture.mechanic.ts
// engine/leagues/crucible.mechanic.ts
// ...
```

**What this gives us:**

- **New league = new plugin file.** No core engine changes unless the mechanic's pattern is genuinely novel.
- **Plugin isolation.** Tincture bugs don't break Determination aura.
- **Optional loading.** Client only loads mechanics for leagues the user cares about. Server loads all.
- **Data-driven baseline.** If the mechanic fits existing patterns (new gem, new keystone, new unique), no plugin needed — just data.
- **Plugin = documentation.** Reading `tincture.mechanic.ts` tells a new developer exactly what Tinctures did in 3.25.

**80/20 rule formalized:**

| Category | Data-Driven | Plugin |
|----------|-------------|--------|
| New unique item with existing mod types | ✓ | — |
| New keystone that sets stat to value | ✓ | — |
| New gem with existing damage types | ✓ | — |
| New flask that grants existing buffs | ✓ | — |
| New mechanic that modifies skill behavior (Tinctures) | — | Plugin |
| New damage type (Soul Damage?) | — | Plugin + registry update |
| New aura interaction pattern (Link skills?) | — | Plugin |
| New enemy mechanic (Sentinel empowerment?) | — | Plugin |

## 11. Final Domain Model

### 11.1 Stat Registry

```
Game Data (GGG files)
  │  auto-generated
  ▼
GameStatIds (numeric IDs, 2200+ entries)
  │  mapping file (architect-maintained)
  ▼
StatRegistry (domain keys, ~200 entries)
  │  grouped by category
  ▼
StatKey (typed accessor, metadata-carrying)
```

### 11.2 Modifier Lifecycle

```
Text Mod ("+99 to maximum life")
  │  ModParser
  ▼
ModifierDefinition (statKey, type, value, conditions, multipliers, effects)
  │  BuildFactory
  ▼
ModifierInstance (definition + source tracking)
  │  Added to ModDBSnapshot
  ▼
[Pipeline evaluates conditions, resolves effects, handles fixpoint]
  │
  ▼
AppliedModifier (instance + computed contribution + trace)
```

### 11.3 Effect Types (NEW)

```
ModifierDefinition
  ├── stats: StatEntry[]           (direct stat contributions)
  ├── conditions: ConditionExpr    (boolean gate)
  ├── multipliers: ScaleExpr       (numeric scale)
  └── effects: ModifierEffect[]    (cascading effects)
        ├── GrantBuffEffect        (buff → more mods)
        ├── GrantAuraEffect        (aura → more mods)
        ├── GrantSkillEffect       (trigger skills)
        ├── ConvertStatEffect      (stat conversion)
        └── OverrideStatEffect     (stat override)
```

### 11.4 Condition System (EXPANDED)

```
Effectiveness = Gate × Scale

Gate (ConditionExpr):
  ├── AND / OR / NOT
  ├── StateCondition        ("is Moving")
  ├── ActorCondition        ("enemy is Burning")
  ├── SkillTagCondition     ("is Attack")
  ├── SlotCondition         ("socketed in Helmet")
  ├── RecentlyCondition     ("Killed Recently")
  ├── TimerCondition        ("every 4 seconds")
  └── DuringActionCondition ("while casting")

Scale (ScaleExpr):
  ├── ConstantScale         (no scaling)
  ├── ChargeScale           (× frenzy charges)
  ├── StatScale             (× stat / per)
  ├── StackScale            (× poison count)
  ├── DistanceScale         (× distance factor)
  └── ManaSpentScale        (× mana spent recently)
```

## 12. Final Calculation Pipeline

### Phase dependency graph (computed, not hardcoded)

```
Pass 1: Static ModDB Build (no dependencies on runtime state)
  ├── BaseStats (class, level, game constants)
  ├── Items (equipped gear → mods)
  ├── Tree (passive allocations → mods)
  ├── Skills (gem data → skill instances)
  └── Config (enemy settings, buffs, charges)

Pass 2: Effect Resolution (FIXPOINT — iterate until stable)
  ├── AuraEffectMods (modifiers that scale auras)
  ├── AuraGrants (auras grant mods; scaled by aura effect)
  ├── BuffGrants (conditional buffs: "on kill", "on hit")
  ├── CurseEffectMods (modifiers that scale curses)
  ├── CurseGrants (curses applied to enemies; priority-based slots)
  └── ConvertStatEffects (Iron Reflexes, CI, MoM, etc.)

Pass 3: Enemy State (runs once after Pass 2 is stable)
  ├── FlaskGrants (flask effect application)
  ├── NonDamagingAilments (shock, chill, scorch → enemyDB)
  └── Exposure (fire/cold/lightning exposure → enemyDB)

Pass 4: Calculation (runs once)
  ├── Defence (armour, evasion, resistances, block, recovery)
  ├── DefenceEstimations (EHP, max hit taken, MoM split, guard)
  ├── Triggers (trigger skill DPS aggregation)
  └── Offence (conversion → crit → penetration → mitigation → ailments → DPS)
```

### Fixpoint convergence

```typescript
function resolveEffects(snapshot: ModDBSnapshot): ModDBSnapshot {
  let current = snapshot;
  let iteration = 0;

  while (iteration < 10) {
    const effects = collectActiveEffects(current);  // all effects from active mods
    if (effects.length === 0) break;

    const newMods = effects.flatMap(e => e.resolve(current));
    if (newMods.length === 0) break;  // no new mods → stable

    current = current.withMods(newMods);
    iteration++;

    // Safety: if the same effect fires twice with the same output, we're stable
    if (current.structuralHash === previousHash) break;
  }

  return current;
}
```

## 15. Rules for AI Coding Agents

These rules apply to ALL AI agents (you are one) working on this codebase:

| # | Rule |
|---|------|
| R1 | **Never use string stat names.** Always `S['defense.life']` or typed accessor from StatRegistry. Compiler errors are better than runtime bugs. |
| R2 | **Never switch on stat name or mod type.** Use `StatKey.aggregation.kind` or `ModType` discriminated union. Add cases to the union, not to the switch. |
| R3 | **Never mutate a ModDBSnapshot.** `addMod` returns a new snapshot. Mutable state is scoped to `CalcEnv.queryCache` only, and cleared between phases. |
| R4 | **Every formula function must accept and return plain typed objects.** No `this`, no class state, no closures over mutable data. |
| R5 | **Every computed number is a `CalculatedValue<T>` with a trace node reference.** No bare `number` return from calculation functions. |
| R6 | **Traces are flat DAGs.** `inputs: string[]` (IDs), never nested objects. One trace node = one calculation. |
| R7 | **Pipeline phases declare reads/writes/effects.** The scheduler computes order. Never hardcode a phase order. |
| R8 | **New league mechanics are plugins in `engine/leagues/`.** Core engine changes only for genuinely novel patterns. |
| R9 | **Integration tests compare old engine output to new engine output at every phase.** Gate merges on `oldOutput === newOutput` (within tolerance). |
| R10 | **Never import Electron, `fs`, `path`, or `window` in `@helper/poe-engine`.** Engine is isomorphic. All platform access is injected. |
| R11 | **Test data is committed.** `data/test/builds/` contains real PoB XML exports. Tests use real builds, not hand-crafted mock data. |
| R12 | **PoB is the reference.** If our engine produces X and PoB produces Y for the same build: we have a bug. File it, fix it, add a regression test. |
